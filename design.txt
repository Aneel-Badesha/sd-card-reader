File system: FatFS vs littlefs

FatFS (File Allocation Table):
    -industry standard for SD card
    -optimized for the 512 byte block architecture of SD cards
    -not fail-safe, if power is lost during update whole system can become corrupted
    -use FAT if you need to swap the card with a computer

littlefs:
    -optimized for high integrity logging
    -uses copy-on-write where old data is never overwritten until the new data is safely sotred
    -immune to powerloss corruption
    -cannt read it on a PC easily
    -performance drops for large files
    -use littlefs if the card stays permanently inside a device and must never "brick" itself due to a power outage


SD Card Interface: SDMMC vs SPI
SDMMC:
    -SDMMC is the native SD protocol designed specifically for high-speed memory access
    -supports 4-bit mode, offering up to 4x the bandwidth of SPI
    -features a dedicated hardware controller that manages DMA transfers with minimal CPU overhead
    -supports advanced SD card functions and higher clock speeds (up to 40-50 MHz)

    -on the ESP32, you are restricted to specific IO_MUX pins (GPIO 14, 15, 2, 4, 12, 13)
    -equires external pull-up resistors (typically 10kΩ) on most lines for stable high-speed operation
    -eMMC chips cannot be used over SPI and must use SDMMC

SPI:
    -SPI is a generic serial protocol that allows the SD card to act as a standard SPI peripheral
    -can be routed to any GPIO pin via the ESP32's GPIO matrix
    -easily shares the bus with other SPI devices using a dedicated Chip Select (CS) pin
    -lower hardware requirements; often easier to debug for basic data logging
    -restricted to 1-bit mode, making it significantly slower for large file transfers
    -every block read/write requires more command/response cycles compared to native SDMMC
    -in the ESP-IDF framework, SPI mode can exhibit higher latency due to software-managed CS and protocol encapsulation

-Decided to use SPI with FAT32

Project Challenges:
    -SD card needs a 10kΩ pullup resistor to prevent MISO from floating
    -SD card needs 5V not 3.3V input because it has an onboard 5V->3.3V regulator
    -FAT 8.4 format limits filenames to 8chars, allocated a small heap buffer for longer filenames
